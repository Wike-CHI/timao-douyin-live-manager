## 目标
- 核查当前并发相关配置并给出是否“高并发”结论
- 提供无侵入的优化建议（提升请求并发、连接池、线程使用）

## 现状概览
- Web 进程：`uvicorn --workers 1`，PM2 `instances: 1`（并发较低）
- 数据库池：MySQL `pool_size=50`、`max_overflow=20`（较高并发友好）
- Redis 池：`REDIS_MAX_CONNECTIONS` 可通过环境变量设置，当前为 `50`
- 计算线程：`OMP_NUM_THREADS=4`、`MKL_NUM_THREADS=4`（控制数值库并行）
- 未发现仓库内 Nginx/K8s HPA 配置

## 建议改动
1. 提升 Web 并发
   - 将 PM2 的启动参数中的 `--workers` 提升到 `2-4`（按 CPU 核心数评估）
   - 维持 PM2 `exec_mode: 'fork'`，避免 Python 下的 cluster 模式
2. 保持/校准连接池
   - MySQL：保持 `pool_size=50`、`max_overflow=20`，观察高峰期是否需上调
   - Redis：通过环境变量确保 `REDIS_MAX_CONNECTIONS>=50` 与业务匹配
3. 线程与阻塞任务
   - 保持 CPU 线程数不与 `workers` 过度叠加，避免过度上下文切换
   - 阻塞任务继续通过 `run_in_executor/to_thread` 下沉到线程池
4. 部署层（如需）
   - 若有入口代理，外部配置 Nginx（`worker_processes`、`worker_connections`）或容器副本扩缩

## 验证步骤
- 压测：以 `wrk`/`ab` 验证吞吐和 P95/P99 延迟
- 监控：观察 MySQL/Redis 连接占用与溢出，以及进程 CPU/内存
- 回滚策略：若资源占用过高，将 `--workers` 回调至安全值

## 风险与注意
- 增加 `workers` 会提升内存占用；含大模型加载时需评估单进程内存
- 线程与进程并行叠加可能导致争用；以真实负载下的观测为准
