# 后端端口连接问题排查指南

## 问题描述

`npm run dev` 启动后，出现以下错误：
```
[ERROR] FastAPI主服务 健康检查失败: HTTPConnectionPool(host='127.0.0.1', port=8080): 
Max retries exceeded with url: /health (Caused by NewConnectionError(...): 
Failed to establish a new connection: [WinError 10061] 由于目标计算机积极拒绝，无法连接。')
```

## 审查人

叶维哲

## 问题原因分析

### 1. 端口配置问题

**配置位置**：
- `server/.env`：`BACKEND_PORT=8080`
- 默认端口（代码中）：`11111`

**可能原因**：
- `.env` 文件未被正确加载
- 环境变量被其他配置覆盖
- 端口冲突导致服务启动失败

### 2. FastAPI 启动时间问题

FastAPI 启动需要：
- 连接数据库（MySQL）
- 初始化 Redis 缓存
- 加载 AI 模型（如果配置了）
- 启动 WebSocket 服务

**原设置**：等待 5 秒
**问题**：数据库连接慢时，5 秒可能不够

### 3. 编码问题

Windows 系统下，UTF-8 编码可能导致：
- 日志乱码
- 进程通信异常
- 启动脚本执行失败

## 解决方案

### 方案 1: 使用诊断脚本（推荐）

运行诊断脚本检查问题：

```bash
node scripts/诊断与排障/check-backend-port.js
```

诊断脚本会检查：
1. 端口 8080 占用情况
2. Python 进程状态
3. 健康检查端点连通性

### 方案 2: 手动检查端口

**Windows:**
```cmd
netstat -ano | findstr ":8080"
```

**查看进程详情:**
```cmd
tasklist /FI "PID eq <PID号>"
```

**Linux/Mac:**
```bash
lsof -i :8080
```

### 方案 3: 手动启动 FastAPI

尝试直接启动 FastAPI，查看详细错误：

```bash
# 激活虚拟环境
.venv\Scripts\activate

# 手动启动 FastAPI
python -m uvicorn server.app.main:app --host 127.0.0.1 --port 8080 --log-level debug
```

观察启动过程中的错误信息。

### 方案 4: 检查数据库连接

数据库连接失败会导致 FastAPI 无法启动：

```python
# 测试数据库连接
python -c "from server.app.database import init_database; from server.config import config_manager; init_database(config_manager.config.database); print('数据库连接成功')"
```

### 方案 5: 检查日志文件

查看详细的启动日志：

```bash
# 查看服务管理器日志
cat logs/service_manager.log

# 查看最近的日志
tail -n 100 logs/service_manager.log
```

## 常见问题

### Q1: 端口 8080 被其他程序占用

**解决方案**：
1. 修改 `server/.env` 中的 `BACKEND_PORT`
2. 或杀死占用端口的程序

### Q2: 数据库连接失败

**症状**：
- FastAPI 进程启动但立即退出
- 日志中有数据库连接错误

**解决方案**：
1. 检查 `server/.env` 中的数据库配置
2. 确保 MySQL 服务正在运行
3. 测试数据库连接

### Q3: 防火墙阻止连接

**症状**：
- 端口被占用但无法连接
- 外部程序无法访问

**解决方案**：
1. 临时关闭防火墙测试
2. 添加防火墙规则允许 8080 端口

### Q4: Python 虚拟环境问题

**症状**：
- 找不到模块
- 启动命令失败

**解决方案**：
```bash
# 重新创建虚拟环境
python -m venv .venv

# 激活虚拟环境
.venv\Scripts\activate

# 重新安装依赖
pip install -r requirements.txt
```

## 预防措施

### 1. 端口冲突检查

在 `package.json` 的 `dev` 脚本中已添加端口清理：
```json
"dev": "npm run kill:ports && concurrently ..."
```

### 2. 自动重试机制

服务管理器会自动重启失败的服务（最多重试一次）。

### 3. 日志监控

所有服务日志都会保存到 `logs/` 目录：
- `logs/service_manager.log` - 服务管理器日志
- 可用于事后分析
