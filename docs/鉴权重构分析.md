# 鉴权模式重构分析

## 当前问题

### 1. 鉴权逻辑过于复杂

**当前实现**：
```python
# auth.py: get_current_user
- HTTPBearer(auto_error=False) # 从Header获取
- Query(None) # 从URL参数获取
- JWT验证
- Session token验证（fallback）
- 多层try-except
```

**问题**：
1. ✗ 两种token验证方式（JWT + session token），增加复杂度
2. ✗ 多层异常处理，难以调试
3. ✗ Token传递方式不统一（Header vs Query）

### 2. 前端token传递不一致

**当前状态**：
- ✓ 普通API请求：使用Authorization header
- ✗ SSE流（/api/ai/live/stream）：使用query参数
- ✗ WebSocket：token传递方式不明确

**结果**：
- JWT验证失败：`401: Could not validate credentials`
- AI分析流断开，主播画像数据不完整

### 3. 数据流问题

**当前状态**：
- ✓ 直播弹幕：正常
- ✗ 语音转写：未启动（转写记录：0条）
- ✗ AI分析：数据传输不完整
  - 只有"语气：专业陪伴"
  - 缺少：人物、节奏、用词、氛围分数等

## 重构方案

### 原则

1. **KISS原则**：保持简单，避免不必要的复杂性
2. **奥卡姆剃刀**：如无必要，勿增实体
3. **统一性**：所有地方使用相同的鉴权方式

### 简化策略

#### 1. 只保留JWT验证

```python
# ✓ 简化后
- 只使用JWT token
- 去掉session token验证
- 统一token验证逻辑
```

#### 2. 统一token传递方式

```python
# ✓ 优先级
1. Authorization header（标准方式）
2. Query参数（SSE和WebSocket的备选）
```

#### 3. 简化异常处理

```python
# ✓ 简化
- 一个验证函数
- 清晰的错误信息
- 简单的try-except
```
