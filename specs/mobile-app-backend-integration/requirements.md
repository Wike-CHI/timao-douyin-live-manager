# 移动端应用与后端服务集成需求文档

## 介绍

本需求文档描述如何将提猫直播助手的后端服务封装部署到服务器，并使用 React Native 构建跨平台移动端应用（iOS + Android）。移动端应用将提供直播监控、弹幕管理、AI分析、数据统计等核心功能，与现有的桌面端应用形成完整的产品生态。

---

## 需求

### 需求 1 - 后端服务服务器部署

**用户故事：** 作为系统管理员，我需要将 FastAPI 后端服务部署到云服务器上，以便移动端和其他客户端能够远程访问 API 服务。

#### 验收标准

1. While 后端服务已开发完成，when 执行部署流程时，系统应当支持将 FastAPI 应用部署到 Linux 服务器（Ubuntu/CentOS）上。
2. When 部署完成后，系统应当通过 HTTPS 协议提供 API 服务，端口可配置（默认 443）。
3. When 访问健康检查端点 `/health` 时，系统应当返回服务状态信息（healthy/unhealthy）。
4. When 服务器重启时，系统应当自动启动后端服务（使用 systemd 或 supervisor）。
5. When 服务异常崩溃时，系统应当自动重启服务进程。
6. While 存在高并发请求，系统应当通过负载均衡（Nginx）分发请求到多个后端实例。
7. When 配置环境变量时，系统应当从 `.env` 文件或系统环境变量中读取配置（数据库、Redis、AI服务等）。
8. When 访问 API 文档时，系统应当在 `/docs` 路径提供 Swagger UI 接口文档。

#### 功能点
- 支持 Docker 容器化部署
- 支持 Nginx 反向代理配置
- 支持 SSL/TLS 证书配置（Let's Encrypt）
- 支持日志轮转和管理
- 支持数据库迁移脚本
- 支持 Redis 连接池配置
- 支持 CORS 跨域配置（允许移动端域名）
- 支持 API 限流和防护（防止 DDoS）

### 需求 2 - 移动端应用架构设计

**用户故事：** 作为产品经理，我需要使用 React Native 构建跨平台移动端应用，支持 iOS 和 Android 平台，提供与桌面端一致的核心功能体验。

#### 验收标准

1. When 使用 React Native 框架开发时，系统应当支持 iOS 和 Android 双平台代码复用率 > 90%。
2. When 应用启动时，系统应当显示启动页（Splash Screen）和应用标识。
3. When 用户未登录时，系统应当跳转到登录页面。
4. When 用户已登录时，系统应当自动使用 JWT Token 进行 API 认证。
5. When Token 过期时，系统应当自动使用 Refresh Token 刷新访问令牌。
6. When 网络请求失败时，系统应当显示友好的错误提示（网络错误、服务器错误、业务错误）。
7. When 应用切换到后台时，系统应当保持 WebSocket 连接或支持后台任务。
8. When 应用进入前台时，系统应当自动刷新最新数据。

#### 功能点
- 使用 React Native CLI 或 Expo 作为开发框架
- 使用 TypeScript 进行类型安全开发
- 使用 React Navigation 进行页面路由管理
- 使用 Redux Toolkit 或 Zustand 进行状态管理
- 使用 Axios 或 Fetch 进行 API 调用
- 使用 AsyncStorage 存储用户 Token 和配置
- 使用 React Native WebSocket 进行实时数据推送
- 支持深色模式（Dark Mode）
- 支持推送通知（Firebase Cloud Messaging / Apple Push Notification）
- 支持离线缓存和同步

### 需求 3 - 用户认证与授权

**用户故事：** 作为移动端用户，我需要使用用户名/邮箱和密码登录应用，登录后可以访问个人订阅信息和功能权限。

#### 验收标准

1. When 用户输入用户名/邮箱和密码时，系统应当调用 `/api/auth/login` 接口进行认证。
2. When 登录成功时，系统应当保存 Access Token 和 Refresh Token 到本地存储。
3. When 调用需要认证的 API 时，系统应当在请求头中携带 `Authorization: Bearer <token>`。
4. When Token 即将过期（剩余时间 < 5分钟）时，系统应当自动刷新 Token。
5. When 用户选择"记住我"时，系统应当保存 Token 的有效期延长（Access Token 8小时，Refresh Token 10年）。
6. When 用户未选择"记住我"时，系统应当使用短期 Token（Access Token 1小时，Refresh Token 1天）。
7. When 用户退出登录时，系统应当清除本地 Token 并调用 `/api/auth/logout` 接口。
8. When 用户注册新账户时，系统应当调用 `/api/auth/register` 接口并验证邮箱。

### 需求 4 - 直播监控功能

**用户故事：** 作为主播或助手，我需要在移动端查看实时直播弹幕、礼物、观众数等数据，并通过悬浮窗接收 AI 生成的分析和话术建议。

#### 验收标准

1. When 用户进入直播监控页面时，系统应当调用 `/api/douyin/start` 接口启动直播监控。
2. When 监控启动成功时，系统应当通过 WebSocket 或 SSE 接收实时弹幕数据。
3. When 接收到新弹幕时，系统应当在列表中实时显示弹幕内容、用户昵称、时间戳。
4. When 接收到礼物时，系统应当显示礼物名称、数量、总价值。
5. When 系统接收到 AI 分析结果时，系统应当通过悬浮窗形式弹出显示分析内容（不显示转录字幕）。
6. When 系统接收到 AI 话术建议时，系统应当通过悬浮窗形式弹出显示话术内容。
7. When 用户点击悬浮窗时，系统应当展开显示完整内容，支持一键复制话术。
8. When 用户拖拽悬浮窗时，系统应当允许用户调整悬浮窗位置（屏幕边缘吸附）。
9. When 用户暂停/恢复监控时，系统应当调用相应的 API 接口更新监控状态。
10. When 用户结束监控时，系统应当调用 `/api/douyin/stop` 接口停止监控并保存数据。
11. When 直播监控页面显示时，系统应当不显示语音转录字幕内容。

#### 功能点
- 支持悬浮窗组件（Floating Window / Overlay）
- AI 分析和话术生成结果通过悬浮窗展示
- 悬浮窗支持展开/收起、拖拽、位置记忆
- 隐藏语音转录字幕功能（移动端特有）
- 悬浮窗支持一键复制话术内容

### 需求 5 - 直播数据统计与复盘

**用户故事：** 作为主播，我需要在移动端查看历史直播的数据统计，包括弹幕数量、观众峰值、礼物总额、AI分析报告等。我可以手动点击"生成复盘"按钮来生成完整的复盘报告。

#### 验收标准

1. When 用户进入数据统计页面时，系统应当调用 `/api/live_report/list` 接口获取历史直播记录。
2. When 用户选择日期范围时，系统应当根据时间范围过滤直播记录。
3. When 用户点击某条直播记录时，系统应当调用 `/api/live_report/{id}` 接口获取详细报告。
4. When 显示统计图表时，系统应当使用 React Native 图表库（如 Victory Native、react-native-chart-kit）渲染数据。
5. When 直播记录中显示复盘状态时，系统应当显示"已生成"或"未生成"状态。
6. When 用户点击"生成复盘"按钮时，系统应当调用 `/api/live_review/generate` 接口手动触发复盘生成。
7. When 复盘生成时，系统应当生成完整的复盘内容（前期版本不按会员类型限制，所有用户都使用完整复盘能力）。
8. When 复盘生成中时，系统应当显示加载状态和进度提示。
9. When 复盘生成完成后，系统应当推送通知并显示复盘内容。
10. When 用户查看复盘报告时，系统应当显示完整的复盘内容（包含摘要、统计数据、图表、AI分析、热词、深度分析和建议）。
11. When 用户导出报告时，系统应当支持导出为 PDF 或分享链接。
12. When 数据加载失败时，系统应当显示错误提示并提供重试按钮。

#### 功能点
- **手动触发复盘生成**（用户点击按钮）
- **完整复盘能力**（前期版本所有用户都使用完整复盘，不区分会员类型）
- 复盘生成状态追踪和通知
- 复盘内容包含所有功能（摘要、统计、图表、AI分析、热词、深度分析、建议）

#### 备注（未来功能）
- **按会员类型自动生成复盘**：此功能暂时不实现，但已记录需求。未来版本中，直播结束后可自动根据会员类型（免费/基础/高级/企业）生成对应级别的复盘。
  - 免费会员：基础复盘（文本摘要）
  - 基础会员：标准复盘（含数据统计）
  - 高级会员：详细复盘（含AI分析）
  - 企业会员：完整复盘（含深度分析和建议）

### 需求 6 - 订阅与支付管理

**用户故事：** 作为用户，我需要在移动端查看当前的订阅套餐、支付记录，并可以升级或续费订阅。

#### 验收标准

1. When 用户进入订阅页面时，系统应当调用 `/api/subscription/current` 接口获取当前订阅信息。
2. When 显示套餐列表时，系统应当调用 `/api/subscription/plans` 接口获取所有可用套餐。
3. When 用户选择套餐并点击支付时，系统应当调用 `/api/payment/create` 接口创建支付订单。
4. When 支付方式为移动端支付（支付宝/微信）时，系统应当调用相应的 SDK 唤起支付界面。
5. When 支付成功时，系统应当调用 `/api/payment/callback` 接口确认支付并更新订阅状态。
6. When 用户查看支付记录时，系统应当调用 `/api/payment/list` 接口显示历史支付记录。
7. When 订阅即将到期时，系统应当推送通知提醒用户续费。
8. When 用户取消订阅时，系统应当调用 `/api/subscription/cancel` 接口取消订阅。

### 需求 7 - 设置与配置

**用户故事：** 作为用户，我需要在移动端修改个人资料、通知设置、API 配置等个性化选项。

#### 验收标准

1. When 用户进入设置页面时，系统应当显示个人资料、通知设置、API 配置、关于等选项。
2. When 用户修改个人资料时，系统应当调用 `/api/auth/profile` 接口更新用户信息。
3. When 用户修改密码时，系统应当调用 `/api/auth/change-password` 接口更新密码。
4. When 用户配置抖音直播房间号时，系统应当调用 `/api/douyin/config` 接口更新配置。
5. When 用户配置 AI 服务时，系统应当调用 `/api/ai_gateway_api/config` 接口更新 AI 网关配置。
6. When 用户开启/关闭推送通知时，系统应当保存设置并注册/取消注册推送令牌。
7. When 用户查看版本信息时，系统应当显示应用版本号和更新日志。

---

### 需求 8 - 实时通知与推送

**用户故事：** 作为用户，我需要在移动端接收实时通知，包括新弹幕提醒、AI 话术建议、直播异常警告等。

#### 验收标准

1. When 用户允许推送通知时，系统应当向服务器注册推送令牌（FCM Token / APNs Token）。
2. When 服务器发送推送通知时，系统应当在前台和后台都能接收并显示通知。
3. When 用户点击通知时，系统应当跳转到相应的页面（弹幕详情、AI建议等）。
4. When 用户关闭应用时，系统应当支持后台推送通知（需要配置后台任务）。
5. When 用户禁用推送通知时，系统应当取消注册推送令牌。

## 非功能需求

### 1. 性能要求
- API 响应时间 < 500ms（正常网络环境）
- 页面加载时间 < 2秒
- 图片加载支持懒加载和缓存
- 支持分页加载，避免一次性加载大量数据

### 2. 安全性
- 所有 API 请求使用 HTTPS
- Token 存储在安全的本地存储中（Keychain / Keystore）
- 支持证书固定（Certificate Pinning）防止中间人攻击
- 敏感数据（密码、Token）不记录到日志
- 支持生物识别登录（指纹/面部识别）

### 3. 用户体验
- 支持深色模式
- 支持多语言（中文、英文）
- 界面适配不同屏幕尺寸（iPhone、iPad、Android 各种尺寸）
- 流畅的页面转场动画
- 友好的错误提示和空状态
- 支持下拉刷新和上拉加载

### 4. 兼容性
- iOS: 支持 iOS 13.0 及以上版本
- Android: 支持 Android 8.0 (API 26) 及以上版本
- 支持 iPhone SE、iPhone 12/13/14/15 系列、iPad
- 支持主流 Android 设备（华为、小米、OPPO、vivo、三星等）

### 5. 可维护性
- 代码遵循统一的编码规范（ESLint、Prettier）
- 使用 TypeScript 进行类型检查
- 组件化和模块化设计
- 完善的错误处理和日志记录
- 单元测试覆盖率 > 60%

## 设计风格和配色

### UI 设计风格
- **设计语言**：Material Design 3（Android）和 iOS Human Interface Guidelines（iOS）
- **组件库**：React Native Paper（Material Design）或 React Native Elements
- **字体**：系统默认字体（San Francisco / Roboto）

### 配色方案
- **主色调**：
  - 主色：`#667eea`（紫色，品牌色）
  - 辅助色：`#764ba2`（深紫色）
- **功能色**：
  - 成功：`#10b981`（绿色）
  - 警告：`#f59e0b`（橙色）
  - 错误：`#ef4444`（红色）
  - 信息：`#3b82f6`（蓝色）
- **中性色**：
  - 背景：`#ffffff`（浅色模式）/ `#121212`（深色模式）
  - 文本：`#1f2937`（浅色模式）/ `#f9fafb`（深色模式）
  - 边框：`#e5e7eb`（浅色模式）/ `#374151`（深色模式）
