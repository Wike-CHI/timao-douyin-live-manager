# 需求文档 - API 一致性问题修复（中优先级）

## 介绍

本文档定义了 API 接口一致性检查中发现的 12 个中优先级问题的修复需求。这些问题虽然不会导致功能完全失效，但会影响功能完整性、数据准确性和用户体验。修复这些问题将提升系统的健壮性和可维护性。

### 问题概览

| 模块 | 问题数 | 主要问题类型 |
|------|--------|--------------|
| 认证模块 | 2 | 字段缺失 |
| 订阅模块 | 3 | 类型不匹配、字段缺失 |
| 支付模块 | 1 | 字段缺失 |
| 抖音模块 | 1 | 字段缺失 |
| 音频转写模块 | 3 | 字段缺失、类型定义不完整 |
| 其他模块 | 3 | 验证缺失、类型统一 |

---

## 需求

### 需求 1 - 认证模块字段完善

**用户故事**: 作为开发者，我希望前端能够访问后端返回的所有用户认证相关信息，以便实现完整的用户状态管理和 AI 功能控制。

**问题 ID**: AUTH-001, AUTH-002

#### 验收标准

1. When 用户成功登录时，前端 LoginResponse 接口应当包含 `firstFreeUsed` 字段（布尔类型），用于标识用户是否已使用首次免费额度
2. When 用户成功登录时，前端 LoginResponse 接口应当包含 `aiUsage` 字段（对象类型），包含以下子字段：
   - `tokens_used`: 已使用的 token 数量（数字）
   - `token_quota`: token 配额（数字或 null）
   - `token_limit`: token 限制（数字或 null）
   - `requests_used`: 已使用的请求次数（数字）
   - `request_limit`: 请求限制（数字或 null）
   - `first_free_used`: 是否使用首次免费（布尔）
3. When 前端调用登录 API 时，应当能够正确解析并存储这些字段到认证状态中
4. When 前端需要判断 AI 功能可用性时，应当能够通过 `aiUsage` 字段获取准确的配额信息

---

### 需求 2 - 订阅模块类型一致性

**用户故事**: 作为开发者，我希望订阅相关的数据类型在前后端保持一致，避免数据转换错误和精度丢失。

**问题 ID**: SUB-001, SUB-002, SUB-003

#### 验收标准

1. When 前端接收订阅计划数据时，`price` 字段应当使用字符串类型而非 number，避免浮点数精度问题
2. When 前端获取当前订阅信息时，响应应当包含完整的 `plan` 嵌套对象，而非仅 `plan_name` 字符串
3. When 前端接收订阅计划的 `duration` 信息时，应当明确定义字段名称和格式：
   - 后端使用 `duration_days` (整数)
   - 前端需要定义清晰的转换规则（如 "30天", "365天"）
4. When 前端进行金额计算时，应当使用专门的金额处理工具类，确保计算精度

---

### 需求 3 - 支付模块功能完善

**用户故事**: 作为用户，我希望在创建订阅时能够使用试用期功能，享受完整的订阅体验。

**问题 ID**: PAY-003

#### 验收标准

1. When 前端调用创建订阅 API 时，应当支持传递 `trial_days` 参数（可选，整数）
2. When 用户选择订阅计划时，界面应当显示试用期信息（如果该计划支持试用）
3. When 创建订阅请求包含 `trial_days` 参数时，后端应当正确处理并返回包含试用期信息的订阅
4. When 订阅包含试用期时，前端应当在订阅详情中显示试用结束日期

---

### 需求 4 - 抖音模块功能完善

**用户故事**: 作为主播，我希望能够监控需要登录才能访问的直播间，扩大监控范围。

**问题 ID**: DY-001

#### 验收标准

1. When 前端启动抖音监控时，应当支持传递可选的 `cookie` 参数（字符串）
2. When 用户配置了抖音账号 Cookie 时，前端应当将 Cookie 传递给后端
3. When 后端接收到 Cookie 参数时，应当使用该 Cookie 访问抖音直播间
4. When Cookie 无效或过期时，系统应当给出明确的错误提示

---

### 需求 5 - 音频转写模块类型完善

**用户故事**: 作为开发者，我希望音频转写模块的接口定义清晰完整，便于调试和维护。

**问题 ID**: AUDIO-001, AUDIO-002, AUDIO-004

#### 验收标准

1. When 前端配置 VAD 参数时，应当有明确的类型定义和默认值说明：
   - `vad_min_silence_sec`: 最小静音时长（秒，浮点数，可选）
   - `vad_min_speech_sec`: 最小语音时长（秒，浮点数，可选）
   - `vad_hangover_sec`: 语音结束后的延迟（秒，浮点数，可选）
   - `vad_rms`: 音量阈值（浮点数，可选）
2. When 前端获取音频转写状态时，响应应当包含 `model` 字段（字符串），标识当前使用的模型
3. When 接收 WebSocket 消息时，应当有明确的消息类型定义：
   - `transcription`: 转写结果消息
   - `level`: 音量级别消息
   - `status`: 状态变更消息
   - `error`: 错误消息
4. When 处理不同类型的 WebSocket 消息时，应当有类型安全的处理逻辑

---

### 需求 6 - 全局输入验证

**用户故事**: 作为用户，我希望在提交表单前就能发现输入错误，避免无效请求导致的等待和错误提示。

**问题 ID**: MISC-001

#### 验收标准

1. When 用户输入邮箱地址时，前端应当验证邮箱格式的有效性（与后端 Pydantic EmailStr 验证器一致）
2. When 用户输入手机号码时，前端应当验证手机号格式（与后端验证规则一致）
3. When 用户输入数值字段时，前端应当验证范围和类型（与后端 Field 约束一致）
4. When 用户输入字符串字段时，前端应当验证长度限制（与后端 min_length/max_length 一致）
5. When 验证失败时，应当显示明确的错误提示，说明正确的输入格式
6. When 表单包含验证错误时，提交按钮应当禁用，避免无效请求

---

### 需求 7 - 日期时间类型统一

**用户故事**: 作为开发者，我希望日期时间字段有统一的格式和处理方式，便于前后端数据交换。

**问题 ID**: MISC-002

#### 验收标准

1. When 后端返回 `datetime` 类型字段时，应当序列化为 ISO 8601 格式字符串（如 "2025-11-02T19:38:10.123Z"）
2. When 前端接收日期时间字符串时，应当使用统一的类型定义（DateTimeString）
3. When 前端需要显示日期时间时，应当使用统一的格式化工具函数
4. When 前端需要比较日期时间时，应当先转换为 Date 对象
5. When API 文档中描述日期时间字段时，应当明确标注格式为 ISO 8601

---

### 需求 8 - 错误响应统一

**用户故事**: 作为开发者，我希望错误响应有统一的格式和处理方式，便于错误处理和用户提示。

**问题 ID**: MISC-003

#### 验收标准

1. When 后端返回错误响应时，应当使用统一的格式：`{ detail: string }` 或 `{ detail: { message: string, code?: string } }`
2. When 前端接收错误响应时，应当有统一的解析逻辑，能够处理：
   - 简单字符串格式：`{ detail: "错误信息" }`
   - 对象格式：`{ detail: { message: "错误信息", code: "ERROR_CODE" } }`
   - 验证错误格式：`{ detail: [{ loc: [...], msg: "...", type: "..." }] }`
3. When 前端显示错误信息时，应当提取正确的错误文本并友好展示
4. When 错误包含错误代码时，应当根据错误代码提供特定的处理逻辑（如重试、跳转等）
5. When 网络错误发生时，应当有统一的处理逻辑和用户提示

## 非功能性需求

### 性能要求

1. 字段添加不应增加 API 响应时间
2. 前端验证应在 100ms 内完成
3. 类型定义不应增加打包体积超过 5KB

### 兼容性要求

1. 保持向后兼容，已有功能不受影响
2. 新增字段应为可选，避免破坏现有代码
3. 类型修改应使用类型别名，便于渐进式迁移

### 可维护性要求

1. 类型定义应集中管理，避免重复
2. 验证逻辑应可复用，提取为工具函数
3. 文档应同步更新，包括 JSDoc 和 API 文档

---

## 约束条件

### 技术约束

1. 必须保持与现有 TypeScript 版本兼容（4.9+）
2. 必须保持与现有 Python 版本兼容（3.11+）
3. 不引入新的外部依赖（除非必要）

### 业务约束

1. 修复不应影响现有用户的使用
2. 修复应在 2 个工作日内完成
3. 所有修复必须有单元测试覆盖

---

## 优先级排序

按实施顺序：

1. **第一批（核心数据）**：
   - 需求 2：订阅模块类型一致性（避免精度问题）
   - 需求 7：日期时间类型统一（影响多个模块）
   - 需求 8：错误响应统一（影响用户体验）

2. **第二批（功能完善）**：
   - 需求 1：认证模块字段完善
   - 需求 3：支付模块功能完善
   - 需求 5：音频转写模块类型完善

3. **第三批（增强功能）**：
   - 需求 4：抖音模块功能完善
   - 需求 6：全局输入验证
